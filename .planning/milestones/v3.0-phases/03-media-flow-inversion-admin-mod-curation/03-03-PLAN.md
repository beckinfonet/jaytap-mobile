<objective>
Lock in MEDIA-07's invariant: NO listing reaches `live` status without at least one photo. Two enforcement points are required because M2 has TWO paths to `live`:

1. **POST /api/moderation/properties/:id/approve** (existing M2 mod-action handler at `moderationRoutes.js:94-133`) — direct mod approval.
2. **PUT /api/moderation/listings/:id** (M2 MOD-14 edit-on-behalf, current handler at `moderationRoutes.js:445-617`) — mod edits and flips to `live` atomically.

Both paths must reject with HTTP 400 + `code: 'MEDIA_REQUIRED'` when `media.photos.length === 0`. Per D-09, the gate is route-level fetch+check BEFORE the atomic `findOneAndUpdate` — NOT a Mongoose pre-save hook (would lose race-safety) and NOT a Mongoose schema-required (would block valid empty-photo `pending` listings).

Purpose: This is the trust boundary that makes the entire phase work. Plan 03-12 client UX (Approve button disabled when photos empty) is defense-in-depth UX; this plan is the actual security gate. RESEARCH §Pitfall 2 documents the false-negative race semantics (acceptable; mod re-taps).

Output: 2 surgical edits to existing handlers + ~4 new supertest describe blocks.
</objective>
