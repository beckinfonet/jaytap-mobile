## Goal

Replace M1's single-screen `CreateListingScreen` with a 6-step contextual listing creation flow (every screen and every field determined by previous answers) per `.planning/phases/999.1-contextual-listing-flow-m3-anchor/SPEC.md`, and invert the media flow so admin/mod uploads photos / videos / 3D-tour after metadata submission. Closes 2 M2 carry-forward UX/data-integrity bugs (ROLE-11 frontend popup-recovery + Phase 4.5 landlord-app uid-mismatch).

## Hard Rules Carried Forward

- No Firebase SDK in either repo (RN client OR backend) — REPO RULE per memory `no-firebase-sdk.md`. Backend continues using `jose` for JWKS verification.
- No `react-navigation` migration — custom `App.tsx` state machine stays. New 6-step flow mounts as overlay or screen-stack within the existing pattern.
- No Firebase custom claims as role source — MongoDB `userType` remains authoritative.
- M1's 3-category 9-type taxonomy preserved — CLAUDE.md guard. SPEC's flat 5-type list reframed (apartment + house → Residential; office + commercial → Commercial; hotel + hostel → Hospitality, kept split per tour-first UI).
- M2's status enum preserved (`pending | live | rejected | archived`). SPEC's `draft | pending_moderation | published | rejected` reframed 1:1 cosmetically only.
- EN+RU bilingual parity for every new UI string (CI gate enforces).
- Manual physical-device QA on iPhone 15 Pro Max + Moto G XT2513V.
- Geographic scope: KG launch market (Bishkek). KZ + UZ expansion remains M4+ — KZT + UZS currencies deferred.

## Decisions Promoted to PROJECT.md

(Captured as Key Decisions rows in PROJECT.md at milestone close.)

- **Nested Mongoose schema (location/basics/conditionAndAmenities/content/terms/media)** — M2's flat shape didn't scale to per-property-type conditional sub-fields cleanly. Nested shape aligns with the 6-step flow's natural data partitioning. Migration pattern reused from M2 Plan 02-02.
- **Status enum stays unchanged** — SPEC's `draft | pending_moderation | published | rejected` reframed 1:1 cosmetic only onto M2's `pending | live | rejected | archived`. Avoids dual-enum reconciliation. Media-pending state handled as queue-side filter, not new enum value.
- **6-step contextual flow as overlay (custom `App.tsx` state machine)** — No `react-navigation` migration. New flow mounts via existing overlay pattern; M2 `moderatorContext` prop carries forward verbatim.
- **Media flow inversion (admin/mod uploads post-submission)** — Users submit metadata only; admin/mod curates media via new mod-queue surface. Backend MEDIA_REQUIRED gate blocks approval until ≥1 photo. Pre-existing M1+M2 user-uploaded media survives migration verbatim. User-side multer surface deleted; AWS IAM rotation reinterpreted as API-surface removal per memory `aws-iam-jaytap-prod-s3.md` (the only path from RN client → S3 PutObject is gone).
- **Atomic deletion of CreateListingScreen + CreateListingForm/ barrel** — Plan 02-09 deleted 13 files / 3449 LOC in 3 atomic commits with sentinel `check-create-listing-screen-removed.sh`. Admin doc-verification surface preserved by extraction into standalone `AdminVerificationScreen.tsx` (zero CreateListingForm dependency).
- **`useModActionGuard` hook + 11 handler wires** — CARRY-01 closed via shared hook with `is403PermissionError(err)` matcher. 11 mod-action submit handlers route 403s through `resetLoading + closeModal + refreshRole`; banner auto-surfaces; bespoke per-screen `Alert.alert` + `onBack` patterns dropped from RoleManagementScreen.
- **Anti-spoofing grep gate carried forward (3rd consecutive milestone)** — `scripts/check-no-landlord-uid-spoofing.sh` joins M2's actoruid + Phase 3's media-stripped sentinels in `npm test` chain. All audit-row writes source uid from JWKS-verified `req.firebaseUid`. Survived 3 milestones without regression.
- **M1 D-02 pattern fires again at Plan 05-07** — Play Console rejected Android versionCode 31; reactive bump to versionCode 32 + versionName 3.0.0 → 3.0.1. Memory `release-android-versioncode-trust-play-console.md` captures the pattern (now fired twice: M1 versionCode 25→28; M3 versionCode 31→32 + versionName 3.0.0→3.0.1). Always check Play Console history before declaring Android baseline. M4 backlog has the runbook item.
