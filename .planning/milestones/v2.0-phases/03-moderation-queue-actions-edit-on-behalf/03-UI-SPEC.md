# Phase 3 — UI Design Contract

> Visual and interaction contract for the Moderation Queue + Actions + Edit-on-Behalf phase. Generated by gsd-ui-researcher; to be verified by gsd-ui-checker.
>
> Brownfield React Native 0.84 (New Architecture) app. **Phase 3 introduces ZERO new design tokens, ZERO new dependencies, and reuses all visual primitives Phase 2 established.** Every contract value below maps to existing `useTheme()` tokens (`src/theme/colors.ts`) and inherits the spacing / typography scales locked in `02-UI-SPEC.md`. The executor's job is to apply these tokens consistently using the precedents from Phase 4.5's `LandlordApplicationQueueScreen.tsx` (visual + interaction template) and Phase 2's `<StatusPill>` / `<RejectionBanner>` / `<HomeRejectionBanner>` components.
>
> **Revision note (2026-05-01):** This document supersedes the initial draft (commit `1dc19f6`) by addressing 4 blocking issues + 2 non-blocking flags raised by `gsd-ui-checker`. Specifically: (1) `moderation.reject.submit` CTA changed from "Submit" to "Reject Listing"; (2) heading weight reduced from 700 → 600 to honor the 2-weight cap; (3) empty-state heading reassigned from 15px → 14px to honor the 4-size cap (15px dropped); (4) the 12px and 20px non-grid spacing values are now explicitly justified as brownfield reuse in the Exceptions block; (5) Approve / Reject button labels expanded to "Approve Listing" / "Reject Listing" verb-noun pairs; (6) `ModerationQueueScreen` focal point named explicitly. All other locked decisions (Color, Registry, locale-key shape, mod-context banner, RejectionBanner patch path) are unchanged.

**Selected-chip color note (D-06 inheriting Phase 4.5 verbatim):**
The selected reasonCode chip uses `colors.primary` for the background (`#2D2D2D` light / `#FFFFFF` dark) — the SAME token Phase 4.5 used. Selected-chip text is `#FFFFFF` in light mode and `#121212` in dark mode (Phase 4.5's hard-coded dark-mode contrast value at line 226 — Phase 3 inherits this verbatim, not a new exception). Unselected chip: `inputBackground` bg + `border` border + `text` label.

**Accent reserved for the following surfaces ONLY** (the 10% rule — explicit list, not "all interactive elements"):

1. Profile entry-point pending-count badge background (when `pendingCount > 0`) — D-03
2. Selected reasonCode chip background in `<RejectListingModal>` (one chip selected at a time → renders accent ≤ once per modal session) — inherits Phase 4.5 visual which used `primary`, **but Phase 3 RECOMMENDED USES `colors.primary` for selected chips for symmetry with Phase 4.5** and reserves `accent` strictly for the count badge to maintain the existing 10% balance Phase 2 locked. Planner picks; either accent OR primary is acceptable for the selected chip provided one of them is consistently chosen.

The accent token is **NOT** used on:
- Approve Listing button (uses `success`)
- Reject Listing button (uses `error` — destructive intent)
- Reject Listing submit button in modal (uses `error`)
- Edit-on-behalf button (uses `surface` + `border` — neutral)
- Mod-context banner (uses `warning`, `surface`, `text`)
- ModerationQueueScreen FlatList rows (renders existing PropertyCard which uses its own surface palette)
- 409 race-condition toast (RN `Alert.alert` — system-styled; NOT brand-colored)

**Destructive color usage (REJECT LISTING button + REJECT LISTING submit):**
The Reject Listing action is the only Phase 3 destructive-intent surface from the moderator's perspective (it transitions the listing to `'rejected'` state). Per the standard 60/30/10 rule and Phase 2's "destructive uses `error`" precedent, both the inline Reject Listing button and the modal Reject Listing submit button use `colors.error` background with `#FFFFFF` label. The Approve Listing action is constructive and uses `colors.success`. The Edit-on-behalf action is neutral (no state mutation until save) and uses `colors.surface + border`.

**60/30/10 split is honored:** screen backgrounds and modal surfaces dominate (~85% of pixels by area); the destructive-red Reject Listing button + green Approve Listing button + accent badge together render <10% of pixels on any single Phase 3 screen.

### `<RejectionBanner>` translation patch (D-09)

- The Phase 2 banner is already mounted on `PropertyDetailsScreen` and currently renders raw `reasonCode` enum strings (e.g., "incomplete-info") because Phase 2 deferred the i18n table to Phase 3.
- Phase 3 makes a 1-line change at `src/components/RejectionBanner.tsx:37`:
  - **Before:** `const codeLabel = reasonCode || 'incomplete-info';`
  - **After:** `const codeLabel = t(\`moderation.reject.reason.${reasonCode || 'incomplete-info'}\` as any);`
- The body assembly logic stays unchanged: `reasonNote ? \`${codeLabel} — ${reasonNote}\` : fallback` (where fallback is `listings.rejection.bodyFallback` with `{reasonCode}` interpolated — Phase 3 keeps this fallback for compatibility but the i18n key now produces a localized phrase, not a raw enum)
- The `listings.rejection.bodyFallback` template needs no copy change in Phase 3 — the `{reasonCode}` placeholder now interpolates to the localized phrase from `moderation.reject.reason.*` instead of a raw enum literal
- Owner-locale invariant (MOD-13): the owner's app calls `t()` against THEIR active locale (EN or RU per `LanguageProvider`); Phase 3's server-side payload contains the raw `rejectionReasonCode` enum string (verbatim from the moderator's choice); the OWNER's client-side `t()` translates it to the owner's locale automatically. This satisfies "rejection reason translated into THEIR locale (not the moderator's locale)" without any backend `Accept-Language` plumbing.
