# Phase 2 — UI Design Contract

> Visual and interaction contract for the listing lifecycle status field absorption phase. Generated by gsd-ui-researcher, verified by gsd-ui-checker.
>
> Brownfield React Native 0.84 (New Architecture) app. No design system migration is in scope. Every contract value below maps to **existing** `useTheme()` tokens or established patterns sampled from `RoleRefreshBanner.tsx`, `LandlordApplicationStatusBanner.tsx`, and `PropertyCard.tsx`. The executor's job is to apply these tokens consistently — not to introduce new ones.

---

## Design System

| Property | Value |
|----------|-------|
| Tool | none (React Native — no shadcn / no Tailwind / no component library) |
| Preset | not applicable |
| Component library | none — `StyleSheet.create` + `useTheme()` only (`src/theme/colors.ts` + `src/theme/ThemeContext.tsx`) |
| Icon library | `lucide-react-native` (project-wide convention; CONVENTIONS.md §Import Organization) |
| Font | Platform default (`-apple-system` on iOS / `Roboto` on Android via RN). Branded serif stack `Platform.select({ ios: 'Georgia', android: 'serif' })` is reserved for marquee text on PropertyCard and is NOT introduced for any Phase 2 surface. |

**No new dependencies in this phase.** No design-token additions. Phase 2 is the absorb-into-existing-tokens phase.

## Spacing Scale

Declared values (multiples of 4; matches existing JayTap convention sampled from `RoleRefreshBanner.tsx` `paddingVertical: 12, paddingHorizontal: 16` and `PropertyCard.tsx` `borderRadius: 18 / 20 / 24`):

| Token | Value | Usage in Phase 2 |
|-------|-------|------------------|
| xs | 4px | Pill internal vertical padding; gap between pill icon (if any) and label |
| sm | 8px | Gap between status pill and PropertyCard title row; inner padding on `<RejectionBanner>` chevron |
| md | 12px | `<RejectionBanner>` + `<HomeRejectionBanner>` vertical padding (matches existing `RoleRefreshBanner.banner.paddingVertical: 12`) |
| md+ | 16px | `<RejectionBanner>` + `<HomeRejectionBanner>` horizontal padding (matches existing `RoleRefreshBanner.banner.paddingHorizontal: 16`); segmented-control horizontal margin from screen edges |
| lg | 24px | OwnerListings empty-state vertical padding above CTA; gap between segmented control and FlatList |
| xl | 32px | OwnerListings empty-state vertical breathing room above the empty-text block |

**Pill geometry (status badges on PropertyCard, MOD-07):**
- Border radius: `12px` (capsule-feel; consistent with the existing `borderRadius: 8/12/18` cluster on PropertyCard chips)
- Internal padding: `paddingVertical: 4, paddingHorizontal: 8`
- Position: top-LEFT of the PropertyCard image area, vertically below the existing rent/sale type badge, with `6px` gap between the badges (D-19 supersedes UI-SPEC original — top-right collides with the existing share + heart actions cluster on PropertyCard at the top-right)

**Segmented control geometry (OwnerListings 4-tab control, MOD-06):**
- Total height: `40px` (touch-target minimum honored; renders inside a parent ScrollView header)
- Per-tab horizontal padding: `16px`
- Bottom margin from FlatList: `16px`
- Each tab honors RN minimum 44×44 hit-slop via `hitSlop={{ top: 8, bottom: 8, left: 8, right: 8 }}` (CONVENTIONS.md §Accessibility — `TAP = 44` constant convention)

**Exceptions:** None. All values resolve to multiples of 4. The `12px` pill radius and `12px` banner vertical padding are documented as reuses of the existing `RoleRefreshBanner` and `LandlordApplicationStatusBanner` shapes, not new exceptions.

## Typography

Existing JayTap convention is RN inline styles (no fontSize tokens). Phase 2 declares 4 sizes + 2 weights, all already present in the codebase (sampled from `RoleRefreshBanner.tsx` lines 91-92 and `LandlordApplicationStatusBanner.tsx` lines 160-161):

| Role | Size | Weight | Line Height | Phase 2 Usage |
|------|------|--------|-------------|---------------|
| Pill label | 11px | 600 (semibold) | 14px (~1.27) | Status pill on PropertyCard (`Pending review` / `Rejected` / `Unpublished`) — matches existing `PropertyCard.tsx:309-310` `fontSize: 11, fontWeight: '600'` chip pattern |
| Body | 13px | 400 (regular) | 18px (~1.38) | `<RejectionBanner>` body text (reasonNote + reasonCode translation); `<HomeRejectionBanner>` count subtitle — matches existing `LandlordApplicationStatusBanner.tsx:161` `fontSize: 13, lineHeight: 18` |
| Banner / Tab label | 14px | 600 (semibold) | 20px (~1.43) | `<RejectionBanner>` title row "Listing rejected" / «Объявление отклонено»; `<HomeRejectionBanner>` headline; segmented-control tab label — matches existing `RoleRefreshBanner.tsx:91-92` `fontSize: 14, fontWeight: '600'` |
| Empty-state heading | 15px | 600 (semibold) | 20px (~1.33) | OwnerListings per-tab empty-state heading text — uses semibold (not bold) to honor the 2-weights-max contract; the 15px size alone differentiates the heading from the 14px banner label |

**Total: 4 sizes (11 / 13 / 14 / 15), 2 weights: 400 (regular) / 600 (semibold).**

- Body line-height ratio: ~1.38 (13/18) — matches existing JayTap banners; not the canonical 1.5 because RN body sizes are smaller than web defaults and the existing palette already converged at this ratio.
- Heading line-height ratio: ~1.33–1.43 (15/20 and 14/20) — matches existing JayTap pattern.
- No `fontFamily` overrides in any new component. Platform default sans-serif throughout.

## Color

JayTap uses a centralized palette in `src/theme/colors.ts` with parallel `light` and `dark` records. Phase 2 introduces **zero new tokens** — all status-pill colors, banner colors, and CTA colors map to existing tokens. The 60/30/10 split is interpreted in RN-native terms (background → surface → accent) since there is no traditional web sidebar/nav split:

| Role | Light token | Dark token | Usage in Phase 2 |
|------|-------------|------------|------------------|
| Dominant (60%) — `background` | `#F0F2F5` | `#191A1D` | Screen backgrounds (OwnerListings, HomeScreen, PropertyDetailsScreen behind hero) |
| Secondary (30%) — `surface` | `#FFFFFF` | `#25282F` | PropertyCard surface (status pill renders on top of this); `<RejectionBanner>` neutral surface variant; segmented-control background; empty-state container |
| Accent (10%) — `accent` | `#FF385C` | `#FF5C7C` | "Edit & resubmit" / «Исправить» CTA on `<RejectionBanner>`; "Create listing" / «Создать объявление» CTA in Live + Pending empty states; tab-active underline/fill on segmented control |
| Status — pending — `warning` | `#F59E0B` (amber-500) | `#F59E0B` | `pending` status pill background on PropertyCard; `<RejectionBanner>` is NOT a pending state — the warning token is exclusively for the pending pill in this phase. `onWarning` (`#FFFFFF` light / `#0F172A` dark) is the pill text color. |
| Status — rejected — `error` | `#F44336` | `#EF5350` | `rejected` status pill background on PropertyCard; `<RejectionBanner>` left-edge accent stripe (4px wide) on PropertyDetailsScreen; `<HomeRejectionBanner>` left-edge accent stripe |
| Status — archived — `textTertiary` | `#999999` | `#6B6F76` | `archived` status pill background on PropertyCard. **Decision:** reuse `textTertiary` as the "neutral / muted / inactive" token because `colors.ts` does not define a `neutral` or `subtle` token and Phase 2 is forbidden from adding tokens. The `archived` pill is intentionally low-contrast — archived listings should de-emphasize visually. Pill text color: `#FFFFFF` (light) / `colors.text` (`#F5F5F5` dark) — same convention as `activeChipText` on the existing palette. |
| Status — live — *(no pill)* | n/a | n/a | `live` status renders no pill. PropertyCard appears clean. (D-16: "live = no pill rendered.") |
| Destructive — `error` | `#F44336` | `#EF5350` | Reuses the same `error` token as the rejected pill. Phase 2 has NO destructive actions (archive button is Phase 4; hard-delete is Phase 4 admin-only). The "Edit & resubmit" CTA is **constructive**, not destructive, and uses `accent` (above), not `error`. |
| Banner pill text on `error` background | `#FFFFFF` | `#FFFFFF` | White text on the rejected pill in both modes — matches the existing `activeChipText: '#FFFFFF'` pattern in light mode and the existing convention on `RoleRefreshBanner.onWarning` |

**Accent reserved for the following surfaces ONLY** (the 10% rule — explicit list, not "all interactive elements"):

1. "Edit & resubmit" / «Исправить» CTA button on `<RejectionBanner>` (PropertyDetailsScreen) — D-13 / D-15
2. CTA button on the "Live" empty state and "Pending" empty state in OwnerListings ("Create listing" / «Создать объявление») — D-11
3. Active-tab indicator (background fill or 2px underline) on the OwnerListings segmented control — Pending tab is the default landing tab and renders selected on first mount per D-09

The accent token is **NOT** used on:
- Status pills (warning / error / textTertiary instead)
- `<HomeRejectionBanner>` body (uses `error` left-stripe + `surface` background — the persistent banner is informational, not action-prominent; tapping it routes to OwnerListings → Rejected tab via D-15)
- Inactive segmented-control tabs (use `text` for label / `surface` for background)
- Per-tab empty-state HEADING text (uses `text` / `textSecondary`)

**60/30/10 split is honored:** screen backgrounds and PropertyCard surfaces dominate (~90% of pixels by area); the warning/error pill is a small badge (<1% of area); accent CTA renders on at most 1 button at a time (well under 10%).

### `<RejectionBanner>` (PropertyDetailsScreen, D-13, MOD-08) — 3 EN+RU pairs = 6 keys

| Element | EN | RU | Locale key (suggested) |
|---------|----|----|------------------------|
| Banner title | Listing rejected | Объявление отклонено | `listings.rejection.title` |
| Banner body fallback (when `reasonNote` is empty) | Reason: {reasonCode}. Edit your listing and resubmit for review. | Причина: {reasonCode}. Отредактируйте объявление и отправьте на повторную проверку. | `listings.rejection.bodyFallback` |
| CTA button label | Edit & resubmit | Исправить | `listings.rejection.cta` |

The `{reasonCode}` token interpolates a Phase-3-supplied translation table (CONTEXT.md `<deferred>` "MOD-13 reasonCode → owner-locale translation table location"). Phase 2 ships only the banner shell + the `{reasonCode}` slot — Phase 3 supplies the translation map. Until Phase 3 ships, the banner renders the raw `reasonCode` string (e.g. `incomplete-info`); a follow-up locale key per code lands in Phase 3.

The `reasonNote` free-text from the moderator (when present) renders verbatim under the banner body **without translation** (MOD-13: free-text is stored verbatim and displayed as-is).

The dismiss "X" button uses an icon-only `<X>` from `lucide-react-native`, 16×16, with `accessibilityLabel="Dismiss" / "Скрыть"`. Per-session dismiss is in-memory only (D-13).

### `<HomeRejectionBanner>` (HomeScreen, D-14, MOD-09) — 1 pluralized EN+RU pair = 4 keys (singular + plural × 2 locales)

| Element | EN (singular / plural) | RU (singular / plural) | Locale key (suggested) |
|---------|------------------------|-------------------------|------------------------|
| Banner body | You have 1 listing that needs edits. / You have {N} listings that need edits. | У вас 1 объявление требует исправления. / У вас {N} объявлений требуют исправления. | `home.rejection.banner.singular` / `home.rejection.banner.plural` |

RU plural form simplification: the existing JayTap locale convention does not implement Russian's full 3-form plural (1, 2-4, 5+); it ships singular + plural only. Phase 2 inherits this convention and ships 2 RU forms (1 vs N) — matches the existing `t()` helper's capability.

The Home banner has **no manual dismiss** per D-14 (informational, not noisy; auto-dismisses when count returns to 0). Tapping the banner navigates to OwnerListings → Rejected tab via the existing nav-state pattern + `defaultTab='rejected'` prop.

### Open question deferred to planner — submit-button copy (D-specifics)

REQUIREMENTS.md MOD-03 implies CreateListingScreen's submit affordance now means "submit for review," not "publish immediately." UI-SPEC stance:

- **Recommended (consistent with the contract above):** rename to "Submit for review" / «Отправить на модерацию» — adds 2 locale keys, total Phase 2 budget shifts from 8–12 to 10–14 keys (still within the CONTEXT.md band).
- **Alternative (zero new keys):** keep existing "Publish" / «Опубликовать» copy; rely on the new Pending tab as the user's mental model anchor for "where did my submission go."

Planner decides during plan task breakdown. The contract here treats the recommended copy as default; if the planner picks the alternative, the executor skips the +2 keys and the i18n parity gate continues to pass.

### No destructive actions in Phase 2

- No archive button (Phase 4 — ARCH-01..ARCH-05)
- No hard-delete affordance (Phase 4, admin-only)
- No reject button (Phase 3 — MOD-11)
- No approve button (Phase 3 — MOD-11)
- "Edit & resubmit" is constructive (it does not delete, archive, or otherwise destroy data); the auto-flip `'rejected'` → `'pending'` server-side on save is the only state mutation, and that is reversible (a moderator can re-reject).

Hence the **Destructive confirmation** row in the standard template renders as **n/a — no destructive actions in Phase 2** below.

## Interaction Contracts

### Status pill on PropertyCard (MOD-07)

- Renders top-right of the card image area, inset 8px from the top and right edges
- Shape: capsule (borderRadius: 12)
- Shows ONLY when `status !== 'live'` (D-16)
- Reads `(p.status ?? 'live')` — defensive coalesce per D-07
- No tap handler — the pill is informational; tapping the card itself navigates to PropertyDetailsScreen as today
- `accessibilityLabel` reads the pill label verbatim ("Pending review" / "Rejected" / "Unpublished" — translated)

### `<RejectionBanner>` on PropertyDetailsScreen (D-13, MOD-08)

- Renders at the very top of the screen, above the hero image, when `status === 'rejected'` AND `currentUser.uid === property.ownerUid`
- Layout: 4px `error` left-edge stripe + padded content area (12px vertical, 16px horizontal — matches RoleRefreshBanner)
- Content top row: title + dismiss "X" button (right-aligned)
- Content body row: localized reason (`reasonCode` translation slot + `reasonNote` verbatim)
- Content footer row: full-width "Edit & resubmit" / «Исправить» button (accent background, white text, 12px radius)
- Dismiss "X" sets in-memory state `bannerDismissed = true` for the current process lifetime (D-13). Banner re-evaluates on every navigate-to-this-listing render cycle and on AppState 'active' transitions; the dismiss state is per-process, not per-listing-instance — but the per-listing context (the rejected listing's id) is what the in-memory state keys on (so dismissing one rejected listing's banner does not dismiss banners on others).
- "Edit & resubmit" CTA navigates to `CreateListingScreen` in EDIT MODE (D-15) for that specific listing
- On save, the listing's status auto-transitions `'rejected'` → `'pending'` server-side; the banner clears next render

### `<HomeRejectionBanner>` on HomeScreen (D-14, MOD-09)

- Renders below the search bar, above the property list, when `count(myListings where status === 'rejected') > 0`
- Layout: same shape as `<RejectionBanner>` (4px `error` left-edge stripe + padded content) but no dismiss "X" (informational, auto-dismiss only)
- Content: pluralized body text + chevron-right indicator (decorative)
- Tappable wholesale — `TouchableOpacity` wraps the entire banner; tap navigates to OwnerListings → Rejected tab via `defaultTab='rejected'` (D-15)
- Auto-dismisses (rerenders to null) when the rejected count returns to 0 — driven by the upstream property list state, no manual control

### OwnerListings 4-tab segmented control (D-09, D-10, D-11, D-12, MOD-06)

- 4 tabs in this exact order: **Live / Pending / Rejected / Archived** (matches ROADMAP success criteria #2 enumeration; D-09 lock)
- Default selected tab on mount: **Pending** (D-09 — second tab)
- One fetch on mount (`GET /api/properties?ownerUid=<self>` or equivalent — planner confirms) returning all owner statuses; client filters by active tab (D-12)
- Pull-to-refresh on the FlatList refetches the full set
- Each tab's FlatList includes `ListHeaderComponent={<HospitalitySection ownerListings={tabFilteredListings}/>}` (D-10)
- Each tab's FlatList includes a `ListEmptyComponent` rendering the per-tab empty state (D-11)
- Tab switch animation: instant (no slide / no fade) — matches existing JayTap simplicity bar
- Active tab visual: accent-colored 2px underline (or background fill — planner picks; both consistent with the contract)

### `useAppStateRoleRefresh()` hook (D-17)

- Subscribes to RN `AppState` `change` events inside `AuthContext.tsx`
- On `nextAppState === 'active'`, calls `AuthContext.refreshRole()` if the last refresh was >60s ago
- Cooldown is shared with the Phase-1 403 interceptor's refresh path — both consult the same module-scope `lastRefreshAt: number | null` timestamp
- Phase 2 has NO visible UI surface for this hook; the existing Phase-1 `<RoleRefreshBanner>` mounted in App.tsx renders the user-facing banner if and only if the role actually changed across two consecutive `refreshRole()` calls
