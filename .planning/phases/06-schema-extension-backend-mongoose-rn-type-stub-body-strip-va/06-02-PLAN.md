<objective>
Wire the body-strip enforcement (SCHEMA-04) and the route-layer 0.5-step + integer validators (defense-in-depth alongside Plan 01's schema validators per D-06) into the three write paths: POST `/api/properties`, PUT `/api/properties/:id`, and PUT `/api/moderation/listings/:id`. Implementation uses a new shared pure-function helper at `src/utils/stripResidentialOnlyFields.js` (D-07) — three import lines, three call sites — instead of inlining the strip logic in each handler.

Purpose: Hospitality + commercial listings cannot leak a `basics.bedrooms` field into MongoDB even when the client (legacy app, future SDK, or hand-crafted curl) sends one. The strip is SILENT (no 400, no log) per M3 D-13 precedent (D-08), matching the established "extra body keys disappear without ceremony" pattern. The route-layer 400 with `M4_*_INVALID` codes (D-11) gives Phase 7 RN client error handling a clean discrimination target without requiring it to parse Mongoose error messages.

Output:
- 1 new utility file (`stripResidentialOnlyFields.js`, ~25 LOC).
- 3 route handler edits adding 1 import + 1 call + 1 route-layer 400 block each.
- 1 new unit-test file for the helper (~70 LOC, 5 cases minimum).
- ~5 new test cases in `propertyRoutes.test.js` (D-10 cases 3 + 4 + planner-expanded step-0.5/bedrooms-invalid 400s).
- ~2 new test cases in `moderationRoutes.test.js` (D-10 case 5 + a route-layer 400 case for completeness).
- 1 checkpoint task: post-deploy smoke check on Railway against an existing M3 listing.
</objective>
