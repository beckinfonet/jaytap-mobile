<objective>
Wave 4 of Phase 8 — final submission. Build, archive, upload, submit.

This plan ships v1.0.4 to both stores. It is the longest-tail plan in the phase because most steps run through web UIs (App Store Connect, Google Play Console) and Xcode's archive UI which require user credentials and signing approvals. Per D-12 the submission flow is at planner discretion within these constraints:

- **iOS:** produce a Xcode archive under Xcode 26.x → upload to App Store Connect. TestFlight Internal Testing pass before App Store production review is RECOMMENDED but not blocking.
- **Android:** produce a release `.aab` under the existing keystore → upload to Play Console. Internal Testing track pass before promoting to production is RECOMMENDED but not blocking.
- **No new App Privacy questionnaire submission** — v1.0.3 ASC App Privacy + Play Data Safety declarations are the binding declarations and are inherited (D-13 descope record).

Includes a TestFlight history check + conditional bump per D-03: if a v1.0.4 archive has already been uploaded under `CURRENT_PROJECT_VERSION = 21`, the next archive must bump to 22 (Apple rejects duplicate build numbers within the same MARKETING_VERSION). Default expectation is 21 is unused; the bump task is conditional and may be a no-op.

Output: 1 conditional file edit (pbxproj `CURRENT_PROJECT_VERSION = 21` → `22`, ONLY on collision), 1 iOS .ipa upload via Transporter or Xcode Organizer, 1 Android .aab upload via Play Console, 2 store submissions, 1 final phase-exit verification doc, atomic commits where applicable. Phase 8 closes when both stores show v1.0.4 in their respective queues (TestFlight / Internal Testing or production review).
</objective>
