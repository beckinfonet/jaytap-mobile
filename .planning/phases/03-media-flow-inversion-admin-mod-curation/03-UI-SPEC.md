# Phase 3 — UI Design Contract

> Visual and interaction contract for **Media Flow Inversion (Admin/Mod Curation)**. Generated by `gsd-ui-researcher`. Verified by `gsd-ui-checker`.
>
> **Scope reminder:** product decisions are LOCKED in `03-CONTEXT.md` (D-01..D-16) and `03-RESEARCH.md` (Discretion #1..#8). This document fills the **design system gaps** — tokens, spacing rhythm, photo-grid sizing, interaction states, copy, dark/light parity — that those upstream artifacts gesture at but did not pin down. It does NOT re-litigate locked product decisions.
>
> **Out of scope for this UI-SPEC** (per CONTEXT.md `<deferred>`): per-asset captions / alt-text / ordering pin, separate `media.panoramic[]` slot, video transcoding / thumbnail generation, push notifications for media events, bulk multi-select moderation actions, AsyncStorage draft persistence for in-progress mod media curation.

## Spacing Scale

Multiples of 4; values match `commonStyles` tokens already in the codebase (`src/components/ContextualListingFlow/styles.ts`) plus `RejectionBanner.tsx` / `ModerationQueueScreen.tsx` precedent.

| Token | Value | Usage in Phase 3 |
|-------|-------|------------------|
| `xs` | 4px | Icon-to-text gaps in chips and banner CTA; photo-grid delete-X overlay inset |
| `sm` | 8px | Chip-to-chip gap (`commonStyles.chipRow.gap`); banner header row vertical breathing |
| `md` | 12px | Banner inner padding (`RejectionBanner.body.padding`); photo grid tile gap; mod-action footer vertical padding |
| `lg` | 16px | Default screen edge padding; banner horizontal padding; banner `marginBottom` (matches `RejectionBanner` precedent) |
| `xl` | 20px | MediaCurationScreen `scrollContent` horizontal padding (matches `commonStyles.scrollContent` precedent) |
| `2xl` | 24px | Section gap inside MediaCurationScreen (matches `commonStyles.section.marginBottom`) |
| `3xl` | 40px | Empty-state vertical padding (`paddingVertical: 40`) — mirrors `ModerationQueueScreen.styles.emptyContainer` (60px) but shorter for in-screen empty state |

**Touch targets:**
- Chip min height: **36px** (matches `commonStyles.chip.minHeight`).
- Photo-grid delete-X tap zone: **24px** visible × **44px** hit-slop applied (`hitSlop={{ top: 10, bottom: 10, left: 10, right: 10 }}`) — meets iOS 44pt and Android 48dp guidance without enlarging the visible artifact.
- All footer buttons: **50px** height (matches `commonStyles.footerButton.height` precedent).

**Exceptions:**
- Photo-grid tile size is fixed at **3 columns × screen-width-derived** (see §"MediaCurationScreen — photo grid sizing" below) — not a token; uses `Dimensions.get('window').width` minus padding/gaps.

### Header

- Background: `colors.surface`. Bottom border: `1px` `colors.border`.
- Padding: horizontal 20px (matches `commonStyles.headerContainer.paddingHorizontal`), vertical 12px top + 8px bottom.
- Title: `t('moderation.mediaCuration.header.title')` at 16px / 700 / lineHeight 24, `colors.text`.
- Close X: `lucide-react-native` `X` icon, 24px, `colors.text`. `hitSlop` 8px on all sides. `accessibilityLabel={t('moderation.mediaCuration.header.close.a11y')}`. On press → `props.onClose()`.

### Photo grid sizing

- **3 columns** on phone-class screens (matches Apple Photos / Google Photos convention; comfortable touch target without over-shrinking thumbnails).
- Tile width: `(Dimensions.get('window').width - (paddingX * 2) - (gap * 2)) / 3` where `paddingX = 20` and `gap = 8`. On a 390pt iPhone screen, tile = `(390 - 40 - 16) / 3 ≈ 111pt`. On a 360dp Android screen, tile ≈ 101dp.
- Tile aspect ratio: **1:1 square** (cropped via `Image.resizeMode: 'cover'`).
- Tile border radius: **8px** (matches existing photo-display patterns in PropertyDetailsScreen photo galleries).
- Gap between tiles: **8px** (matches `commonStyles.chipRow.gap`).
- Section vertical gap: **24px** (matches `commonStyles.section.marginBottom`).

**Add tile (`+`):**
- Same dimensions as photo tiles. Background: `colors.primaryLight` (light: `#F2EFE9`, dark: `#353941`). Border: `1px` dashed `colors.border`. Center: `Plus` icon 32px in `colors.textSecondary`.
- Tap → invokes `ImagePicker.launchImageLibrary` with the config from `03-RESEARCH.md` Code Example 3 (`mediaType: 'mixed'`, `selectionLimit: 0`, `assetRepresentationMode: 'compatible'`, `quality: 0.85`).
- The add tile is rendered as the LAST item in the photo grid (after all photo and pending-photo tiles). It is HIDDEN when `photos.length + pendingPhotos.length >= 40`.
- A separate add tile is rendered at the end of the videos grid (hidden when `videos.length + pendingVideos.length >= 5`).

### Photo / video tile (rendered)

- `<Image source={{ uri }}>` filling the tile with `resizeMode: 'cover'`, border radius 8px.
- Top-right corner overlay: a 24×24 circular delete-X button — background `rgba(0,0,0,0.55)` (semi-opaque so X is visible regardless of underlying photo), `X` icon at 14px in `#FFFFFF`. Inset 4px from top and right edges. `hitSlop={{ top: 10, bottom: 10, left: 10, right: 10 }}` for 44pt+ effective touch target.
- Top-left corner: pending-asset badge (only for assets in the `pendingPhotos[]` / `pendingVideos[]` queue — see §"Asset state differentiation").
- **Video tiles only:** Center overlay — a 32px `Video` (lucide) icon with a `rgba(0,0,0,0.4)` circular backdrop (40px) — signals "this is a video, not a photo". No first-frame preview attempted (per CONTEXT.md `<deferred>` "Video transcoding / thumbnail generation"); the placeholder icon stands in.

### Video grid

Same sizing rules as photo grid, with the video icon overlay. Maximum **5 videos** per the cap defined in CONTEXT.md D-07 + Discretion #6. Section label: `Videos ({count}/5)` / «Видео ({count}/5)».

**Decision (researcher discretion):** Photos and videos are rendered in **TWO SEPARATE GRIDS** rather than a single mixed grid. Rationale: separating them (a) makes the count-against-cap text honest per category (40 photos vs. 5 videos cap), (b) gives the mod a faster mental model of "what's where", and (c) avoids cap-confusion (mixed grid would need a "you've hit one cap" UX that mod can't differentiate from "all caps OK"). Cost: more vertical scroll. Worth it.

### Tour URL input

- Single-line `<TextInput>` at `commonStyles.input` dimensions (height 50, border 1px `colors.border`, radius 10, paddingHorizontal 12, fontSize 16).
- Placeholder color: `colors.textTertiary` (`#999999` light, `#6B6F76` dark).
- `keyboardType="url"`; `autoCapitalize="none"`; `autoCorrect={false}`.
- Hint line below: 13px / 400 / lineHeight 18 in `colors.textSecondary`.
- **Validation states:**
  - **Empty (default):** no error styling; tour URL is optional.
  - **Filled and valid (starts with `https://` and parses via `URL` constructor):** input border switches to `colors.success`.
  - **Filled and invalid:** input border switches to `colors.error`; error caption renders BELOW the hint line in 12px / 400 / `colors.error`: `t('moderation.mediaCuration.tourUrl.invalid')`.
  - Validation triggers `onBlur`, NOT `onChangeText` (avoids flicker as user types).
- **Persistence:** tourUrl is sent in the next `Save photos` POST as a multipart `tourUrl` field (per Code Example 3); it is NOT persisted as a separate "save" — there is no per-field auto-save.

### Action footer

- Background: `colors.surface`. Top border: `1px` `colors.border`. Padding: vertical 16, horizontal 20. SafeArea-respecting bottom inset (`Math.max(insets.bottom, 12)` — same pattern as PropertyDetailsScreen mod footer).
- Layout: two buttons in a row, gap 12px, each `flex: 1`, height 50.

**Button A — Save photos (secondary):**
- Background: `colors.surface`. Border: `1px` `colors.border`. Border radius: 10.
- Label: `t('moderation.mediaCuration.save.button')` at 16/600/24 in `colors.text`.
- **States:**
  - **Default (idle, has pending media OR has tourUrl change):** enabled.
  - **Disabled (no pending media AND no tourUrl change):** `opacity: 0.5`; `disabled={true}`. Adjacent hint NOT shown for Save (only Approve gets the hint per D-12).
  - **Loading:** show `<ActivityIndicator color={colors.text} size="small" />` in place of label; button stays at `opacity: 1` but `disabled={true}` to block double-taps; replace label with `t('moderation.mediaCuration.save.loading')`.
- On tap → calls service which POSTs FormData; on success → in-app toast `t('moderation.mediaCuration.save.success')` + clear `pendingPhotos[]` / `pendingVideos[]` (server-stored copies replace them in the photo grid via the `result` payload from the response).

**Button B — Approve & publish (primary/success):**
- Background: `colors.success`. Border radius: 10. No border.
- Label: `t('moderation.mediaCuration.approve.button')` at 16/600/24 in `#FFFFFF`.
- **States:**
  - **Enabled (server-saved `media.photos.length > 0`):** full opacity; tappable. NOTE: pending photos that have not yet been Saved DO NOT count — the server is the trust boundary. So the rule is: `isApproveEnabled = listing.media.photos.length > 0` where `listing` is the latest server-fetched copy. Pending-only mods see Approve disabled even with pending tiles in the grid.
  - **Disabled (`listing.media.photos.length === 0`):** `opacity: 0.5`; `disabled={true}`. Hint text rendered in the row BELOW the buttons: 12/400/16 in `colors.textSecondary`, copy = `t('moderation.mediaCuration.approve.disabled.hint')`. Hint padding-top: 8px.
  - **Loading:** `<ActivityIndicator color="#FFF" size="small" />` replaces label.
- On tap → calls existing `PropertyService.approveListing(listingId)`. On success → trigger `props.onApproveSuccess()`; App.tsx closes overlay and refreshes ModerationQueueScreen.
- **Server fallback:** if a stale client somehow tap-and-submits while `listing.media.photos` was deleted concurrently → backend returns HTTP 400 `MEDIA_REQUIRED` → client surfaces toast `t('moderation.mediaCuration.error.mediaRequired')` and refetches the listing.

### Empty state (no server photos AND no pending)

When `listing.media.photos.length === 0` AND `pendingPhotos.length === 0`:
- The photo grid renders ONLY the `+` add tile (no other tiles).
- Above the grid, the section label `Photos (0/40)` still renders.
- BELOW the grid (full screen width, padding xl=20), an empty-state block:
  - Vertical centered, `paddingVertical: 40`.
  - 48px `Image` (lucide) icon in `colors.textTertiary`.
  - Title at 14/600/20 in `colors.text`: `t('moderation.mediaCuration.empty.title')`.
  - Body at 13/400/18 in `colors.textSecondary`: `t('moderation.mediaCuration.empty.body')`.

When the videos grid is empty AND no pending videos: the section label still renders with `(0/5)` and the lone `+` tile. No additional empty-state block under videos (less critical surface; one block at the photos section is enough guidance).

### Asset state differentiation (research §Diff #7 flagged) — pending vs. server

The photo grid renders TWO classes of tiles:

| Class | Source | Visual | Delete behavior |
|-------|--------|--------|-----------------|
| **Server-saved** | URL from `listing.media.photos[]` (loaded from MongoDB via API) | Standard tile (no badge) | Delete X → calls `DELETE /api/moderation/listings/:id/media?url=...&kind=photo` (per Pattern 3); on success the URL is removed from the server array; tile disappears from grid |
| **Pending (not yet saved)** | Local URI from `pendingPhotos[]` populated by ImagePicker callback | Tile with **top-left "Not saved" badge** + dimmed overlay | Delete X → removes locally from `pendingPhotos[]` array (no network call); tile disappears immediately |

**Pending badge styling:**
- Top-left corner of tile, inset 4px from top and left.
- Pill: padding 2px / 6px (vertical / horizontal), border radius 6px, background `colors.warning` (`#F59E0B`), text `colors.onWarning` at 11/600/14, label `t('moderation.mediaCuration.pending.badge')` ("Not saved" / «Не сохранено»).

**Pending dimmed overlay:**
- Full-tile overlay at `rgba(0,0,0,0.18)` to visually distinguish pending from server-saved at a glance even if the badge is missed.

**Ordering in the grid:**
- Server-saved tiles render first (in the order they appear in `listing.media.photos[]`).
- Pending tiles render after server-saved tiles (in pick order).
- The `+` add tile renders last.

**Why this matters:** Server-saved tiles are deletable via authenticated API calls (which can race with another mod and yield 409 → race toast). Pending tiles are local-only and "free" to remove. A mod looking at the grid must instantly know "this photo is live in MongoDB" vs. "this photo will upload on next Save tap". The Phase-2 chip-pattern precedent uses `colors.activeChipBackground` for a binary state; pending-vs-server is also binary, and the warning-tinted badge + overlay is the chosen distinguisher.

### Loading state — upload progress

Researcher discretion (per `<output>` §6): keep it simple.

**Aggregate spinner pattern (chosen):**
- During the Save photos POST, the entire MediaCurationScreen body becomes touch-blocked via a full-screen overlay (`pointerEvents: 'box-none'` on the parent `View`; an inner translucent layer at `rgba(0,0,0,0.3)` blocks scroll).
- Center: `<ActivityIndicator color={colors.accent} size="large" />` + label below: `t('moderation.mediaCuration.save.loading')` ("Uploading…" / «Загрузка…») in 14/500/20 in `#FFFFFF`.

- On success → overlay dismisses; success toast surfaces.
- On error → overlay dismisses; error toast surfaces.

**Per-file progress bars (rejected):** Would require XHR `onUploadProgress` plumbing through `apiClient` (which does not currently expose it), per-file UI rendering, and cancellation. Out of scope for Phase 3 ship. Re-evaluate in M4+ if mods complain about visibility on slow networks.

### Picker invocation states (Discretion §5 — RN file picker)

The `+` add tile and the picker callback handle these states:

| State | Trigger | UX |
|-------|---------|----|
| **Picker unavailable** | `ImagePicker.launchImageLibrary` falsy at runtime | `Alert.alert(t('common.error'), t('moderation.mediaCuration.error.pickerUnavailable'))` — same pattern as `LandlordApplicationScreen.tsx:111` |

## Race-toast (MOD-15 mirror)

**Reuse existing M2 race-toast surface** (`moderation.race.toast` / `moderation.race.title` keys) — no new copy.

### Trigger surfaces in Phase 3

1. **Concurrent media-delete:** Mod A taps delete-X on photo URL `X`; mod B taps delete-X on the same photo URL `X` simultaneously. Backend `findOneAndUpdate` with `$pull: { 'media.photos': X }` is idempotent (race-safe — both succeed without error). However, if the backend handler also asserts "the URL must have existed" (planner discretion), the second call returns HTTP 409 → client renders M2 race-toast.
2. **Concurrent approve-vs-delete:** Mod A taps Approve while mod B deletes the only remaining photo; Mod A's `/approve` returns HTTP 400 `MEDIA_REQUIRED` (NOT 409 — different code, different toast). This is the `mediaRequired` toast (not the race toast). The mod refetches and sees the photos count went to 0 — the listing is back in "needs media" state.
3. **Concurrent approve-vs-approve (M2 carry-forward):** Mod A and mod B both tap Approve simultaneously; backend's atomic `findOneAndUpdate({ status: 'pending' }, { status: 'live' })` filter only matches once — second call returns 409 → M2 race-toast (existing surface, unchanged by Phase 3).

### Toast styling (existing M2 surface)

Top-of-screen banner-style toast at `colors.surface` background, `colors.error` left stripe, title 14/600 + body 13/400. Auto-dismiss after 4 seconds. Tap to dismiss immediately. (Existing M2 implementation; Phase 3 does not modify.)
