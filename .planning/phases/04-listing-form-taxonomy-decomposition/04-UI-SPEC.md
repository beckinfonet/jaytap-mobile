> Visual and interaction contract for the `CreateListingScreen` taxonomy + decomposition phase. Generated by gsd-ui-researcher (2026-04-24). Verified by gsd-ui-checker.
>
> **Scope discipline.** This phase is ~90% "preserve current pixel behavior" and ~10% "design the new category-chip UX + HospitalitySection + hide-PriceSection-for-Hospitality rule." **Radii (preserved):** `borderRadius: 12` (inputs, datepicker, addButton, imagePickerButton, tourItem, featureTag), `borderRadius: 16` (verification card, submitButton, imageItem), `borderRadius: 20` (chip, featureTag wider variant), `borderRadius: 26` (segmentButton), `borderRadius: 30` (segmentedControl shell).

**Touch targets (enforced):**
- `input` height: 50px (exceeds 44pt iOS minimum, matches existing)

- `segmentButton` (rent/sale, draft/live): 44px tall — already compliant
- `submitButton`: 56px — compliant
- `addButton` (+): 50×50 — compliant

**Exceptions:** None introduced in Phase 4. The `removeFeature` × glyph and `removeTourText` × glyph are already wrapped in `TouchableOpacity`; ensure `hitSlop` is applied in each sub-component carved-out to match CONVENTIONS.md existing practice.

## Typography

**Preserve existing — do NOT introduce a new type scale.** Four sizes, two weights. Matches the current `CreateListingScreen.tsx` StyleSheet.

| Role | Size | Weight | Line Height | Usage |
|------|------|--------|-------------|-------|
| Heading (section title) | 16px | 700 | default (platform ~1.2) | `styles.sectionTitle` — every sub-component header |
| Body / Input text | 16px | 400 | default | `styles.input`, `styles.textArea`, `styles.currencyOptionText (600)`, `styles.segmentText (600)` |
| Label | 14px | 500 | default | `styles.label` (above currency / propertyType), `styles.chipText`, `styles.featureText` |
| Hint / Caption | 12px | 400 italic | default | `styles.hint` — placeholder text below inputs, italic |
| Button text | 16–18px | 600–700 | default | `styles.addTourButtonText` (16/600), `styles.submitButtonText` (18/700) |

**Weight vocabulary:** 400 (regular), 500 (medium — labels + chip text), 600 (semibold — button text, segment text), 700 (bold — section titles, submit button, header title). Four weights total — this is already what the existing file uses; Phase 4 does not add a fifth.

**RU-length overflow (MANDATORY for new chips and section headers):**
- Every `chipText` inside `BasicInfoSection`'s category chip row and propertyType chip row MUST set `numberOfLines={1}` AND `ellipsizeMode="tail"` on the inner `<Text>` so Russian "Промышленный" (12 chars vs English "Industrial" 10 chars) does not force the chip to wrap onto a second visual line.
- Category labels ("Residential" → "Жилая недвижимость" is 19 chars vs 11 — BE CAREFUL; see Color & Copywriting below for the decision to use short RU labels).
- `sectionTitle` in every sub-component: no cap (section headers are single-line anyway on 375pt+ screens; if a narrow-device test fails, add `adjustsFontSizeToFit minimumFontScale={0.85}` — defer to QA matrix cells 3 and 5).

**Heading/body line-height:** Do not explicitly set `lineHeight` in new StyleSheets — inherit the platform default (iOS and Android both ship sensible defaults at these sizes). This matches current behavior; introducing explicit values would drift the baseline.

### PriceSection mount rule (Hospitality showcase-only)

**Implementation:** `{category !== 'Hospitality' && <PriceSection … />}` at the orchestrator JSX level. When Hospitality is active, `PriceSection` is NOT MOUNTED — meaning the currency chip row and the price input do not render AT ALL. This is preferred over a "hide with opacity" or an explanatory banner because:

1. Matches PROJECT.md lock: "no per-night price, contact drives offline booking" — UI should make this invisible, not explained-away.
2. Matches RESEARCH §D unmount pattern (Pitfall 4 — data-rot avoidance; Hospitality selection must not leak a stale `price` value into the payload).
3. Matches CONTEXT "no 'upgrade' CTA, no placeholder" language for gated sections.

**Banner copy:** None. If Hospitality is selected and a user had previously entered a price, the `price` and `currency` values remain in `FormBag` state (Phase 5's `buildPayload` filters them out), but are no longer visible. Phase 5 owns the "don't leak price on submit" logic; Phase 4 owns the visual absence.

## Shared Sub-Component Visual Contract

### Section header style (ALL 7 sub-components)

```tsx
<View style={styles.section}>
  <Text style={[styles.sectionTitle, { color: colors.text }]}>{t('<sectionKey>')}</Text>
  {/* … section body */}
</View>
```

- `styles.section`: `marginBottom: 24` (gap-2xl) — creates the inter-section rhythm
- `styles.sectionTitle`: `fontSize: 16`, `fontWeight: '700'`, `marginBottom: 12` (gap-md)
- Color: `colors.text` — NEVER accent, NEVER surface
- NO icon, NO divider line above/below (divider-less rhythm is the house style)

### Spacing rhythm between sections

From bottom of one section to top of the next: `marginBottom: 24` on the outgoing `styles.section`. No `marginTop` on the incoming one. (Additive margins would double the gap to 48 — avoid.)

### Common prop interface (visual expectations)

```typescript
export interface SectionProps {
  values: FormBag;
  onChange: <K extends keyof FormBag>(field: K, value: FormBag[K]) => void;
  errors: FormErrorBag;   // Phase 4: always {} — reserved for Phase 5 error rendering
}
```

**Phase 4 visual expectation around `errors`:** `FormErrorBag` is passed but NO UI is rendered. Fields have no red-border treatment, no per-field error text below inputs. Phase 5 lands the error visualization contract (red border via `colors.error`, 12px error text below input). Phase 4 sections must already forward `errors={errors}` to child inputs so Phase 5 is a drop-in.

### Scroll behavior on section transitions

- **One** `KeyboardAwareScrollView` at the orchestrator level (current line 504) — **no nested scroll views inside any sub-component.** RESEARCH §"Anti-patterns" flags this explicitly: nested scrolls regress Phase 2 keyboard behavior.
- Conditional mounts (Residential/Commercial/Hospitality) cause RN layout to recalculate the scroll content height when category changes; this is correct and expected.
- When the user switches from Residential to Hospitality, the scroll position should NOT auto-jump. Do not call `scrollToOffset` or similar. Let layout settle naturally.

### Keyboard avoidance inside each section

Inherits from orchestrator-level `KeyboardAwareScrollView`. Sub-components do NOT instantiate `KeyboardAvoidingView` individually. Every `TextInput` in every sub-component inherits the scroll-into-view behavior Phase 2 installed. New inputs in `HospitalitySection` (`rooms`, `maxGuests`) and `BasicInfoSection` category chips require ZERO keyboard-handling code.
