<objective>
Apply the photo-required approval gate to the Moderation Queue listing-review path so it matches the behavior of every other entry path into PropertyDetailsScreen.

**Root cause of the gap (verified by investigation, not assumed):**

The photo-gate (NeedsMediaBanner + Approve-disabled hint + disabled Approve button) is implemented inside `PropertyDetailsScreen.tsx` via the predicate `showNeedsMediaBanner = can('approveListings') && status === 'pending' && photoCount === 0` (lines 298-311). It fires correctly for ANY caller that opens PropertyDetailsScreen.

`ModerationQueueScreen.handleRowTap` (lines 308-320) is an outlier: it explicitly DIVERTS zero-photo rows away from PropertyDetailsScreen and into MediaCurationScreen via `onOpenMediaCuration(listingId)`. This was Phase 3 Plan 03-06 D-01 ("needs-media row routes to curation surface"). Result: the mod never reaches PropertyDetailsScreen, so the banner + disabled-Approve UX never runs on this entry path.

Every other entry to PropertyDetailsScreen (HomeScreen tap at App.tsx:206/702, FavoritesScreen, OwnerListingsScreen, RenterListingsScreen, PropertyMap, plus the queue's own `handleViewTour` / `handleViewVideo`) calls `setSelectedProperty(p)` directly and the banner fires correctly.

**Fix:** Unify `handleRowTap` so ALL rows (regardless of photo count) call `onOpenPropertyDetails(listing)`. PropertyDetailsScreen's existing photo-gate handles the zero-photo UX. The "Add photos" CTA inside the NeedsMediaBanner still routes the mod to MediaCurationScreen via the already-wired `onOpenMediaCuration` callback (App.tsx:1010 + PropertyDetailsScreen.tsx:1550) — that chain is preserved, just moved one screen deeper.

Out of scope:
- The inline Approve button on each queue row (already disables for photoCount===0 at line 344, line 367 opacity, line 370 disabled prop — verified working).
- The "Needs media" / "Has media" / "All pending" filter chips (still useful as a sorting affordance; do not remove).
- The MediaCurationScreen overlay itself (still reachable via NeedsMediaBanner CTA from PropertyDetailsScreen).
- The `onOpenMediaCuration` callback prop on `ModerationQueueScreen` — leave it in the props interface for now (no harm, optional, and removing it churns App.tsx). It simply becomes unused at the row-tap site.

Purpose: Close the gap that lets a mod/admin bypass the photo-required UI when entering via Profile → Moderation Queue.
Output: ModerationQueueScreen unified row-tap routing + updated unit test.
</objective>
